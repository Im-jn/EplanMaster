# PDF 页面内容与矢量语法教程

本文面向在本仓库中开发 **PDF 矢量匹配**（如 `element_finder.py`）时的阅读材料，说明：**一页「画出来」的内容在源码里长什么样**，以及 **content stream（内容流）里会出现哪些运算符与代码形态**。

权威定义见 ISO 32000（PDF 规范）；以下为实战摘要。

---

## 1. 一页的矢量/绘图代码在哪里？

- PDF 由 **间接对象**（`编号 代数 obj … endobj`）组成。
- **页面对象** 的类型一般是 `/Type /Page`，其中常有：
  - **`/Contents`**：指向「这一页怎么画」的内容流；可以是单个间接引用，也可以是 **数组**（多段内容按顺序拼接执行）。
  - **`/Resources`**：字体、图像、表单等资源的字典。
- **内容流** 多半是 **`stream … endstream`**，可能被 **`/FlateDecode`** 等压缩；解压后得到 **明文运算符序列**，这才是矢量匹配主要面对的字符串。

同一页的绘图指令也可能藏在 **表单 XObject**（`/Subtype /Form`）里，通过 **`Do`** 被调用——解析时若只看顶层 `/Contents`，需要递归跟进资源里的表单。

---

## 2. Content Stream 的基本语法形态

内容流是一串 **后缀记号（postfix）**：先是操作数，最后是 **运算符名字**（PDF 里多为单个 ASCII 字母，少数两个字母）。

典型片段：

```text
q
1 0 0 1 50 720 cm
0 0 100 50 re
S
Q
```

含义简述：`q`/`Q` 保存/恢复图形状态；`cm` 变换当前矩阵；`re` 建矩形路径；`S` 描边。

常见 **记号类型**：

| 形态 | 含义 |
|------|------|
| 数字 | 整数或浮点数（坐标、矩阵分量等） |
| `/Name` | 名字对象，如 `/F1`、`/Img1` |
| `(…)` / `<…>` | 字符串（文本显示时常出现） |
| `[ … ]` | 数组 |
| `<< … >>` | 字典（较少直接出现在裸内容流中间，多在对象层） |

运算符与操作数之间用 **空白** 分隔即可。

---

## 3. 坐标系与变换（矢量匹配必懂）

- 默认用户空间：**原点通常在页面左下角**，**x 向右、y 向上**（与屏幕左上角原点习惯不同）。
- **`cm`**：`a b c d e f cm`，等价于右乘当前 **CTM（当前变换矩阵）**，六个数为：

  \[
  \begin{bmatrix} a & b \\ c & d \\ e & f \end{bmatrix}
  \]

  变换作用于后续路径与文字。

- **`q` / `Q`**：**图形状态栈**。`q` 入栈，`Q` 出栈；其间对的 **`cm`**、线宽、颜色等会一起恢复。
- 匹配算法若要做「平移无关」比较，通常会把路径整体 **减去 bbox 原点**（本仓库 `element_finder.py` 即类似思路）。

---

## 4. 路径（Path）——矢量图形的主体

路径由「构造」指令画出，再由「描边/填充」指令显现。

### 4.1 当前点与子路径

- 多条 **moveto** 会开始新的 **子路径**；是否闭合由各运算符语义决定。

### 4.2 常用路径构造运算符

与本仓库 `PATH_OPERATORS` 一致的一组核心运算符：

| 运算符 | 操作数 | 含义 |
|--------|--------|------|
| `m` | x y | moveto：移动到 (x,y)，新子路径起点 |
| `l` | x y | lineto：直线到 (x,y) |
| `c` | x1 y1 x2 y2 x3 y3 | 三次贝塞尔曲线（当前点为 p0，到 (x3,y3)） |
| `v` | x2 y2 x3 y3 | 曲线：第一个控制点默认为当前点 |
| `y` | x1 y1 x3 y3 | 曲线：第二个控制点默认为当前点 |
| `re` | x y w h | 矩形：`addrectangle`，常用于边框、背景 |
| `h` | — | closepath：闭合当前子路径 |

矩形也可用 `m l l l h` 等价画出；生成器风格不同会导致 **同一几何形状代码写法不同**，匹配时往往不能只靠字符串相等。

### 4.3 常用描绘（paint）运算符

对应绘制线段或填充封闭区域（本仓库 `PAINT_OPERATORS`）：

| 运算符 | 含义 |
|--------|------|
| `S` | stroke：描边当前路径 |
| `s` | closepath + stroke |
| `f` | fill（非零环绕规则） |
| `F` | 同 `f` |
| `f*` | fill（偶奇规则） |
| `B` | fill + stroke（非零规则） |
| `B*` | fill + stroke（偶奇规则） |
| `b` | closepath + fill + stroke |
| `b*` | closepath + fill（偶奇）+ stroke |
| `n` | 结束路径但不填充不描边（常用于裁剪路径定义前的构造） |

另外还有 **`W` / `W*`**（裁剪路径）等，路径构造完成后、`n` 或 paint 之前可能穿插 **`W`** 设置裁剪。

## 5. 图形状态（影响「线」长什么样）

常与矢量一起看的有：

| 运算符 | 含义 |
|--------|------|
| `w` | 线宽 |
| `J` | 端点样式（butt / round / square） |
| `j` | 连结样式 |
| `M` | 斜接极限 |
| `d` | 虚线样式 dash array + phase |
| `ri` | 渲染意图（多用于图像，有时也出现） |
| `i` | 平直度容差 |

颜色（描边/填充）常用：

| 运算符 | 含义 |
|--------|------|
| `G` / `g` | 灰度 stroke / fill |
| `RG` / `rg` | RGB |
| `K` / `k` | CMYK |

## 6. 页面里还会出现哪些「内容」——不仅是矢量路径？

一页除了 **`m l c … S`**，还经常混合下列类别；做全文索引或匹配时需心里有数。

### 6.1 文本对象（Text Objects）

以 **`BT`**（begin text）与 **`ET`**（end text）包裹，内部常见：

| 运算符 | 含义 |
|--------|------|
| `Tm` | 文本矩阵（六种数字） |
| `Td` / `TD` | 相对位移 |
| `Tf` | 字体与字号：`/F1 12 Tf` |
| `Tj` | 显示字符串 |
| `'`| `'` | 简单定位并显示 |
| `"`| `"`| 字距与显示 |
| `TJ` | 数组形式显示（含 kern） |

矢量算法若只关心几何线段，通常会 **跳过 BT…ET** 块或单独解析。

### 6.2 外部图像与表单（**`Do`**）

| 形态 | 含义 |
|------|------|
| `/ImageName Do` | 绘制图像 XObject（位图），几何由 **`cm`** 等变换决定 |
| `/FormName Do` | 绘制表单 XObject；表单内部又可包含路径、`Do`、文字 |

「矢量 PDF」里的 Logo、渐变栅格化结果常以 **`Do`** 出现——它们 **不是 path 运算符**，但仍占用页面视觉。

### 6.3 内联图像（Inline Image）

介于 **`BI`**（begin inline image）与 **`EI`**（end inline image）之间，中间有 **`ID`**（image data）；关键字 **`BI`**、字典键如 **`/W`**、**/H**、**/ColorSpace** 等。解析二进制段落时需与普通 **`ID`** token 区分语境。

### 6.4 注释与结构化标记（可选）

- **`BDC`** / **`BMC`** … **`EMC`**：标记内容（图层、Tagged PDF、可选实际内容边界）。
- **`BX`** … **`EX`**：兼容性分组。

匹配矢量时可忽略或单独统计。

## 7. 与矢量匹配相关的实务要点

1. **同一几何，多种写法**：矩形可用 **`re`** 或四条 **`l`**；曲线可拆分或改用 **`c`**；顺序一致才可能被本项目当前的序列比对命中。
2. **必须先解压**：`/FlateDecode` 等内容流不解压看到的是二进制，不是运算符文本。
3. **坐标系**：合并 **`cm`** 与 **`q/Q`** 后才能得到最终在页面上的 bbox（简单脚本往往先在当前矩阵下累积点再变换）。
4. **嵌套**：`/Contents` 数组、`Do` 表单、可选 **`Pattern`** 着色等会增加「路径出现的位置」，若要全局匹配需在架构上递归或扁平化。
5. **运算符清单**：ISO 32000 附录 **PDF Operator Summary** 是最全核对表；本文只覆盖页面绘制里最常出现的子集。
